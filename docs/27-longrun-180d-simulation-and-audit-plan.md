## 0. 目標與原則

**三個目標：**

1. **全系統重跑**：在乾淨隔離環境，從 `alembic upgrade head` 建庫開始，以「真實 service／API 流程」模擬門市 **≥180 天**實際營運，涵蓋**每一個模組的每一種資料**（收購、簽署、寄售、銷售、退貨、作廢、採購、進項發票、盤點、購物金、點數、活動、餐飲、現金班別、報表、稽核），不得以 raw-insert 繞過業務邏輯。
2. **全面健檢**：在這份大資料上重新檢視整個系統——帳務不變量、報表口徑、API 正確性、效能（大資料下才現形的慢查詢）、前端 UX 與錯誤——產出分級的優化/改善清單。
3. **手持簽署證據鏈專項**：驗證「手持裝置確認交易＋簽名」功能在大量真實流程下的證據完整性與**可稽核性**，並對簽名檔儲存位置（DB／本地檔案／Cloudflare R2）做出有依據的裁示。

**執行原則：**

- **隔離**：專用 DB `lucamp_sim`＋獨立 port，絕不碰 dev（`lucamp`）與既有 `lucamp_e2e`；pytest 一律 `lucamp_pytest`（見既有踩雷記錄：conftest 會 drop_all）。
- **全流程、不抄捷徑**：模擬資料一律經 service 層或 HTTP API 產生。現有 `seed_large.py` 直接建 `SerializedItem`/`BulkLot` 列、無收購單、無簽署、無現金支出——本計畫的 seed 是**重寫擴充**，不是沿用。
- **Report-first**：發現的問題先分級記錄、附重現證據，經使用者裁示後才修（P0 crash／資料損毀級除外，即時回報並停下）。不邊跑邊改，避免污染基準。
- **可重現**：固定 RNG 種子、記錄 git SHA／alembic head／seed 參數，產出 manifest（筆數、日期跨度、種子），任何結論可綁回資料集重現。
- **修復規範不變**：每個修復獨立短分支、TDD、四道門、Codex review、合併前經使用者確認。
- **優化判準**：沿用「單店單機」校準——真實營運會發生的才修；想像中的多終端併發邊角不做。

## Phase 1 — 模擬資料引擎 `sim_180d`（主要工作量，2–3 天）

新檔 `backend/qa_e2e/sim_180d.py`（＋純函式輔助模組），TDD：日程產生器、金額分佈、狀態機走訪等純邏輯先有單元測試。

### 1.1 時間軸模型（200 天 > 180 天門檻）

- 每個營業日一個 `cash_session`：開帳（零用金 3,000–8,000）→ 全日交易 → 關帳（**照公式實點**，其中 ~5% 天故意有 ±10~100 元差異並記原因——驗差異記錄鏈）。約 **200 個班別**。
- 客流模型：週間 15–30 單/日、週末 40–70 單/日、月中冷月尾熱的波動；近 21 天疊一檔進行中活動（歷史另有 3–4 檔已結束活動）。
- 時區一致採 Asia/Taipei 寫入（驗 UTC 日界在報表上的表現——曾是已知課題）。

### 1.2 資料量與流程覆蓋（全部走真實 service／API）

| 模組 | 量 | 必走流程 |
|---|---|---|
| 會員/客戶 | ~2,500 | 建檔（姓名+手機必填、部分含合法身分證號）、撞號 409、事後補登證號、PII 查看寫稽核抽樣 |
| **收購（買斷）** | ~700 件序號品＋~40 散裝批 | `AcquisitionService` 全流程：定價計算機 margin → **手持切結（K4）**：建 AFFIDAVIT 簽署任務 → 擬真筆跡 PNG 簽名 → 綁定 → 撥款 CASH（現金班別扣款）或 STORE_CREDIT（溢價凍結）；~2% 收購事後作廢（F6.5，含購物金反轉） |
| 寄售 | ~350 件 | 寄售入庫（不簽切結）→ 售出產生 `consignment_settlement` → **付款流程**（PENDING→PAID，走現金班別）；部分因退貨反轉（未付→CANCELLED） |
| **銷售** | **≥4,200 筆** | POS 結帳引擎：序號品/散裝/數量型/餐飲混車；付款 CASH／STORE_CREDIT／MIXED；活動自動折扣；點數累計（餐飲排除）；購物金低消門檻案例；**作廢 ~2%**（點數/購物金/寄售全反轉）、**退貨 ~1%**（API 層，退貨+已開票→折讓路徑於 Amego 抽測內驗證） |
| 簽署任務 | **~900**（AFFIDAVIT ~700＋SCU 購物金扣抵確認＋ACK 交易紀錄簽收） | K2–K5 全鏈：kiosk 角色輪詢→簽名→交回；含少量「簽後未完成→作廢重推」與冪等重放案例 |
| 採購 | ~100 張 PO | 草稿→下單→（部分）收貨→入庫＋stock_movement；**收貨登錄進項發票**（含全空、齊備、事後補登、重複號被擋 4 種案例）；~5 張取消；供應商 ~15 家含 2 家中途停用＋1 家改名（驗名稱快照） |
| 盤點 | ~8 次 | 快照→實點（含有差異/無差異）→ 確認 ADJUST |
| 購物金 | 撥入 ~600 會員、折抵 ~400 筆 | 溢價率期中調整 1 次（驗 premium_rate_history）、餘額不足 409、作廢反轉 |
| 菜單/餐飲 | ~20 品項 | 含改價（寫稽核）、封存後歷史 sale_line 仍可讀 |
| 活動 | 5 檔 | 過去 3–4 檔（activate+end）＋進行中 1 檔；寄售品折扣按比例分攤案例 |
| 設定 | — | 稅率/抽成/低消/溢價各動 1 次（驗設定變更軌跡與快照口徑） |

### 1.3 產出 manifest（fail-closed 證據的落地）

`sim_manifest.json`：各表筆數、日期 min/max、RNG 種子、git SHA、alembic head、執行時長。後續每一層驗證啟動時**先核對 manifest**（筆數/跨度不符即 exit 1）——確保「驗的是這份資料」，不會拿空庫假通過。

## Phase 2 — 六層驗證（2 天）

### Layer A — 四道門（基準，Phase 0 已跑，此處確認未漂移）

### Layer B — 帳本不變量全量掃描（擴充 `longrun_invariants.py`）

既有 13 項全跑，**新增**：

1. 簽署證據完整性：所有 `SIGNED` 任務必有 `signature_image`＋`signed_at`＋冪等鍵；PNG 可解析（同後端驗證子集）；AFFIDAVIT 必綁收購且內容快照與綁定欄位一致。
2. 進項發票守恆：每張 goods_receipt 全空或全備；`net + tax = total`；同店同號無重複。
3. 序號品狀態機：SOLD 不重複售出/入庫；寄售 SOLD 必有 settlement。
4. 散裝批：`remaining_qty ≥ 0`、歸零即 `SOLD_OUT`、扣減總和=售出量。
5. 寄售結算反轉：退貨後無虛假 PENDING/已實現抽成；`reclaim_needed` 正確。
6. 購物金鏈：balance_after 鏈式一致、reversal 配對、凍結溢價=簽署當下值。
7. 現金班別：200 個班別逐一驗 `期望現金 = 開帳 + 銷售現金 − 收購付現 − 寄售付款 ± 調整`，差異筆有原因。
8. 發票狀態：einvoice 關閉期間全部 `NOT_ISSUED` 且銷售完整；點數=floor((total−餐飲)/100) 逐筆重算。
9. 供應商名快照：改名供應商的歷史 PO 仍為原名。

### Layer C — 併發探針（既有 8 項重跑於大資料上）

雙客戶端賣同序號品/散裝超賣/寄售雙付/同冪等鍵/同班別並發/購物金雙花/會員撞號/關帳 vs 收購。

### Layer D — 全 API 掃描

以 `openapi.json` 驅動，對大資料集逐端點打一輪（合法參數＋邊界參數如 naive 日期、超大分頁、空區間）：任何 500 即列 P0/P1；同時記錄**每端點 latency**（見 Layer F）。

### Layer E — 瀏覽器全站走查＋關鍵流程實操

- 全路由（15+）雙 viewport（1366×1000／390×844）截圖、**0 JS console error** 門檻。
- 實操流程（Playwright）：登入→POS 混車結帳（含購物金+活動折扣）→收購含手持簽署（雙瀏覽器：店員端+kiosk 端）→關帳→報表五分頁＋CSV/Excel 匯出→會員中心五分頁→採購建單收貨登發票→盤點確認。
- 報表數字 spot-check：抽 3 個日期區間，報表值 vs SQL 直算比對（毛利、購物金流量、活動成效）。

### Layer F — 效能量測（大資料下建立基準）

- 每個報表/清單端點在 4,200+ 銷售、2,500 會員下的 latency（p50/p95）；>1s 的列出。
- Postgres `pg_stat_statements`／`EXPLAIN ANALYZE` top 10 慢查詢（重點嫌疑：報表 margin_breakdown、trends 逐桶、會員中心跨源合併分頁、庫存清單、FIFO aging）。
- 缺失索引清單（例如以 store_id+日期範圍掃描的大表）。
- 前端：報表頁/會員頁在大資料下的載入時間、bundle 尺寸記錄。
- **產出效能預算基準表**，超標項目列 P2。

## Phase 3 — 手持簽署證據鏈與儲存專項（1 天）

### 3.1 現況（已查核）

- 簽名 PNG 存 **Postgres `signature_tasks.signature_image`（LargeBinary）**，與任務同交易寫入；`SIGNED` 必有簽名+時間戳（DB CHECK）；內容快照 JSONB＋身分指紋內部欄＋冪等指紋。
- 調閱：`GET /signing/tasks/{id}/signature`（staff 權限）回 image/png。
- **初查缺口**：調閱簽名影像**未寫 audit_log**（CLAUDE.md §5 精神：敏感資料查看要留痕）。列入健檢報告驗證與裁示。

### 3.2 稽核演練（在 ~900 筆簽署上）

1. 全量：Layer B 的簽署不變量掃描。
2. 抽樣 20 筆做**完整稽核重建**：從一筆收購/銷售出發 → 找到簽署任務 → 調出簽名 PNG → 核對內容快照（品項/金額/撥款/身分遮罩）→ 核對凍結溢價與帳本分錄 → 確認整鏈在 UI 上店長可自行完成（不需工程師下 SQL）。**若店長無法在 UI 完成調閱，列 P1 改善項**。
3. 邊角：作廢重推的舊任務、冪等重放、簽後收購作廢——證據是否仍可追溯。

### 3.3 儲存方案評估（裁示點 #1）

| 方案 | 優點 | 缺點 | 評估 |
|---|---|---|---|
| **A. 維持 Postgres（建議）** | 與任務同交易、一套備份涵蓋、PII 不出店、900 筆×~20KB≈**18MB/年**（10 年也才 ~200MB，毫無壓力） | DB dump 稍大 | **單店規模最穩** |
| B. 本地檔案系統 | DB 輕 | 雙寫一致性、備份兩套、無交易保證、檔案可被誤刪 | 無收益，不建議 |
| C. Cloudflare R2 主儲存 | 異地、容量無限 | 簽名=個資出店（須端側加密+金鑰管理）、網路依賴（斷網不能簽名？）、新失敗模式、成本與複雜度 | 作主儲存不建議 |
| **A+R2 加密異地備份（可選加購）** | 兼得：本地即時、異地防災 | 需排程與金鑰管理 | 若擔心單機硬碟死亡，**這是正解**：每日 `pg_dump -Fc` 以 age/GPG 加密後上傳 R2（整庫一起備，不只簽名） |

### 3.4 備份還原演練（必做，不論裁示哪案）

`pg_dump -Fc lucamp_sim` → 還原至全新 scratch DB → `alembic current` 核對 → 抽 20 筆簽名 PNG 做 **sha256 比對**（dump 前後一致）→ 產出一頁 runbook（備份指令、還原指令、驗證步驟）。這是簽名「需要時稽核得到」的最後一哩：資料在，還要**災後還原得回來**。
