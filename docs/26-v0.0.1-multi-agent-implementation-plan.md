## 1. 目的與不變量

本計畫把 [25-system-assessment-v0.0.1.md](./25-system-assessment-v0.0.1.md) 的發現拆成可驗證、可獨立 review、盡量不互相衝突的工作。每個 task 都有獨立 branch、worktree、資料庫與測試證據；在 review、回歸測試及使用者確認前，不進入 `main`。

以下是不允許在實作中被犧牲的產品不變量：

1. 門市人員不會因時間到期而在操作途中被自動登出。
2. session 必須可撤銷；停權、降權、登出本裝置或全部裝置應明確生效。
3. 網路暫時中斷不得清除 session 或遺失尚未送出的表單內容。
4. 所有金額使用 `Decimal`，現金與寄售付款必須可稽核、可重送且不重複扣付。
5. 店別資料隔離、權限檢查與後端驗證不可只依賴前端導覽。
6. UI 結論須由真實 backend、PostgreSQL 與 browser 操作／截圖支持。
7. `v0.0.1` 是已存在的基準標籤，不移動、不覆寫；後續改善應使用新版本號。

## 2. 分支、worktree 與代理角色

### 2.1 分支策略

- 文件先留在 `docs/system-assessment-v0.0.1`，確認內容後才 commit。
- 實作開始時，由 root 從 `main@df603fb` 建立 `integration/post-v0.0.1-hardening`。
- 每個 task 從當時已通過驗收的 integration head 建立短生命週期 branch。
- implementer 只在自己的 task branch commit，不得切換 root worktree 的 branch。
- root 是唯一可以建立／移除 worktree、整合 task branch、處理 rebase 與產生總結的人。
- 不直接在 `main` 開發；integration 合併至 `main`、push、移動或新增 tag 都需要使用者明確確認。

### 2.2 四個固定角色

| 角色 | 責任 | 禁止事項 |
|---|---|---|
| Root / orchestration lead | 排程、path lock、建立 worktree、整合、跑跨 task 回歸、彙報進度 | 不跳過 review；不自行推 `main` |
| Implementer A | Backend、migration、Python dependency、金流等 task | 不碰未分配 path；不 push／merge／tag |
| Implementer B | Frontend、QA、自動化、UX task | 不碰未分配 path；不 push／merge／tag |
| Reviewer | 只讀 diff、測試輸出、截圖與風險審查 | 不改碼、不 commit、不替作者修正 |

Reviewer 子代理是額外的獨立檢查，不取代專案規範要求的 Codex review。實際 review 時使用乾淨上下文，只提供需求、base/head SHA、diff、測試輸出與證據，降低受到作者說法影響的機會。

### 2.3 隔離規則

| 資源 | Implementer A | Implementer B |

不得共用既有 `lucamp`、`lucamp_e2e` 資料庫或同一個 browser storage state。任何 task 若新增 migration，該時段只允許一位 implementer 工作。

## 4. 執行 waves

### Wave 0 — Preflight 與基線凍結

由 root 執行，尚不修改產品行為：

1. 保存 `main@df603fb`、`v0.0.1`、目前分支及 dirty files 清單。
2. 確認下列未追蹤檔的去留，不得靜默刪除：
   - `docs/25-system-assessment-v0.0.1.md`
   - `frontend/scripts/fully-e2e-qa.mjs`
   - `frontend/scripts/manual_extra.mjs`
   - `frontend/scripts/manual_shots.mjs`
3. 先 commit 已確認的評估與本計畫文件；一次 commit 只處理文件。
4. 處理 `CLAUDE.md` 引用但實際不存在的 `docs/README.md` 文件落差。
5. 建立 integration branch、兩個 worktree、獨立 DB／port／artifact 目錄。
6. 記錄全套 baseline tests；無法解釋的既有紅燈會阻擋實作。

完成定義：root 可從任一乾淨 worktree 重現相同 base SHA、測試與資料庫版本，且 `main` 未被修改。

DB-1 依序執行：

1. 對使用者指定的長期資料庫做 `pg_dump -Fc`。
2. 還原至全新 scratch DB。
3. scratch DB 執行 `alembic upgrade head`。
4. 驗證稅率回填、採購狀態、簽署資料、health 與 login。
5. 提供備份路徑、restore 指令、資料筆數及驗證結果。
6. 經使用者再次確認，才升級原始長期資料庫。

不可把 `alembic downgrade` 當成此處的可靠 rollback；rollback 證據必須來自成功 restore。

### Wave 3 — AUTH 垂直切片（高風險、序列）

Branch：`feat/revocable-persistent-device-session`

開始前先寫 ADR，確認 auth contract。建議方案：

- 使用者／裝置 session 本身沒有時間到期，不因時間把門市人員登出。
- 短效 access credential 只作內部安全邊界，透過持久化的 server-side device session 無感更新。
- browser 重開後仍可恢復；網路暫時失敗時保留 session 與表單，背景重試。
- 僅在 server 明確回覆「session 已撤銷、帳號停權或權限失效」時要求重新驗證。
- refresh rotation 必須有 grace／replay protocol，避免多分頁或回應遺失把正常使用者登出。
- 優先採 HttpOnly、Secure、SameSite cookie；同時補齊 CSRF 與 CSP 設計。

此設計同時滿足「門市 session 永不因時間過期」與專案對短效 access token／refresh 的安全要求。若最後決定沿用無 `exp` bearer token，必須另寫 ADR 說明為何偏離既有安全規範及如何補償風險，不能直接實作。

實作分三個 checkpoint，但同一時間只有一位 implementer：

1. AUTH-ADR：狀態機、cookie／CSRF、撤銷、refresh replay、多分頁、網路失敗與 migration 設計。
2. AUTH-BE：device session model、migration、login／refresh／revoke API、逐請求停權與降權覆核、contract tests。
3. AUTH-FE：背景更新、browser reopen、本裝置／全部裝置登出、401 分流、角色導覽與真 browser E2E。

完成定義：

- 跨至少兩次 access credential TTL 仍留在原操作流程，不跳登入頁。
- 瀏覽器關閉重開可恢復；暫時離線不清 session。
- 本裝置撤銷、全部撤銷、停權、降權、refresh 重送與兩分頁 race 都有測試。
- 一般店員不顯示管理頁入口，但後端仍拒絕越權請求。
- migration 在空 DB、既有資料 scratch DB 都完成 upgrade／restore 驗證。
- Reviewer 及 Codex adversarial review 無未處理 BLOCKER／HIGH。

### Wave 5 — 現金與寄售（高風險、嚴格序列）

#### CASH-1：`feat/cash-live-reconciliation`

先沿用後端現有 `session_breakdown` 口徑，加入開帳中 summary、movement 明細、實點後預估差異；不在前端另寫一套帳務公式。

若非零差異原因需要 migration，拆成 CASH-2，並與其他 migration 完全串行。

完成定義：expected、收入、收購支出、寄售付款、退款、調整與 close 結果同口徑；非零差異由後端要求原因；重複 close、movement race、rollback、audit 與 Decimal tests 全綠；桌面及手機真實截圖通過。

#### CONS-1：`feat/consignment-worklist`

加入寄售人分組、穩定排序、帳齡、總筆數，以及「全部／目前篩選／本頁」不混淆的合計。以 20、50、200 筆資料驗證分頁和合計。

#### CONS-2：`feat/consignment-batch-payout`

開始前需確認批次收據與 cash movement 呈現。最低要求是同店、同寄售人、固定鎖順序、整批原子成功或 rollback，以及 idempotency key 重送一致。

CONS-2 必須在 CASH 與 CONS-1 後執行，因為三者會碰 OpenAPI 生成物、現金影響或相同寄售 paths。付款／單筆付款／退貨／作廢 race 必須只有一個合法結果，且不能產生重複現金出帳。

## 5. 衝突與 path lock

下列區域是 exclusive ownership；同一時間只允許一個 task 修改：

| Exclusive path／資源 | 原因 |
|---|---|
| `backend/alembic/versions/*` 與任何 migration 操作 | revision、schema 與真實 DB 升級順序不能競爭 |
| `backend/app/core/*` | auth、dependency injection 與全域設定影響面大 |
| `backend/app/shared/enums.py` | 多模組共用 contract |
| `frontend/lib/api.ts`、`token.ts`、`auth.ts` | session 與全站 401 行為 |
| `frontend/app/(authed)/layout.tsx` | responsive shell 與 role nav 衝突 |
| `frontend/app/globals.css` | 幾乎所有頁面受影響 |
| `frontend/openapi.json`、`frontend/lib/api-types.ts` | AUTH → CASH → CONS 嚴格序列並在 rebase 後重生成 |
| 同一 lockfile | dependency resolution 不可由兩個 branch 同時改 |
| `check.sh`、`check.ps1`、`Makefile` | 最終 gate 必須由單一 task 對齊 |

若 implementer 發現工作需要越過 ownership、加入 migration 或修改 exclusive path，立即停止並回報 root 重新排程，不得自行擴張 scope。

## 6. TDD、驗收與 review gate

每個行為變更 task 必須提交下列 evidence packet：

1. task ID、branch、base SHA、head SHA 與需求／不變量。
2. 先失敗的測試或可重現步驟（Red）。
3. 最小實作與通過結果（Green）。
4. changed files 與未碰觸的 exclusive paths。
5. 實際執行的 lint、format、type、unit、integration、build、migration 指令及 pass count。
6. DB 名稱、dataset/run ID、ports、browser viewport、截圖目錄。
7. 已知風險、未涵蓋項目與 rollback 方式。

Reviewer 回報格式固定為：

`Severity｜file:line｜evidence/reproduction｜impact｜recommended fix`

Review 順序：

1. 只讀 reviewer 檢查 scope、測試是否真的先紅後綠、權限／店別／金額／race、UI screenshot。
2. Root 執行專案要求的 Codex review；auth、cash、consignment 使用 adversarial security／concurrency 視角。
3. 只要有 BLOCKER／HIGH，就不整合。
4. 依專案流程，每一輪 review 修正前先向使用者報告 findings 並取得確認；不做無限自動 review-fix loop。
5. 修正由原 implementer 完成，再交同一 reviewer re-review。
6. Task 通過後，root 才整合至 integration branch並跑跨 task regression。

## 7. 立即停止條件

出現以下任一狀況即停止該 task，不 merge、不 push、不 tag：

- agent 位於錯誤 branch／worktree，或修改未分配 path。
- baseline 有無法解釋的紅燈，或測試被刪除、放寬以製造綠燈。
- 新增 migration、修改 shared/core/OpenAPI 卻未先取得 path lock。
- 輸出包含 secrets、真實個資或長期資料庫敏感內容。
- auth 造成任何時間型自動登出、兩分頁 refresh race 或離線時清 session。
- cash／consignment 出現跨店、重複付款、部分成功、不可追查或 Decimal 誤差。
- DB 備份未成功還原，卻準備操作原始長期 DB。
- dependency audit 仍有 high／critical，或為了 audit 強壓不相容版本。
- Reviewer／Codex review 尚有 BLOCKER／HIGH。
- 使用者尚未同意 main merge、push、原始 DB migration 或版本發布。

| 1 | FMT-1 | Implementer A | queued | PRE |
| 1 | QA-1 | Implementer B | queued | PRE |

## 9. 建議的第一個啟動批次

使用者確認本計畫後，第一批只做以下事項：

1. Root 保存並 commit 評估文件與本計畫，不碰產品程式碼。
2. Root 建立 integration branch 與 A／B worktrees，公布 base SHA、DB、ports 與 path ownership。
